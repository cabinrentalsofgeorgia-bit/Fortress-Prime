# Wiki / Knowledge-Plane Audit - 2026-05-06

## Classification

`OPERATIONAL_COGNITION_LAYER_AUDIT`

## Executive Finding

The wiki functions as an operational cognition layer, but it is not yet the canonical operational memory for the current Fortress Legal production system. The app repo has become the execution memory for the newest phases. The wiki remains strongest for product vision, early architecture, RLS/auth decisions, deployment governance, retrieval certification, and production Supabase history.

## What The Wiki Knows

The wiki contains:

- Product/user/role framing.
- Matter/document/entity concepts.
- Early architecture and current-app-state snapshots.
- Auth, authorization, RLS, storage, retention, audit, and external-interaction decisions.
- Context packs that orient future AI/human sessions.
- Production/Supabase/retrieval deployment runbooks.
- Risk registers and legal-domain boundaries.
- Staging/production evidence and controlled activation records.
- Source-of-truth consolidation decision.

## What The App Repo Knows Better

The app repo contains the freshest operational chain:

- Source integrity, remediation, link repair, targeted source completion.
- Limited signoff candidate packet.
- Counsel signoff decision workflow.
- Autonomous learning loop.
- Draft work product generation.
- Feature alignment, hardening, remediation maturity, review operations, review scaling, certification, internal pilot, and human operations.
- Checker, deployment verifier, pilot simulation verifier.
- Runtime rollback artifacts and evidence directories.

## What Exists Only In AI Conversations

Some operational memory is still primarily chat-derived:

- The exact sequence of phase missions and operator intent.
- Why certain branches were stacked rather than merged into canonical base.
- The rationale for using clean phase worktrees when `/home/admin/Fortress-Prime` is dirty.
- The practical habit of using authenticated checker evidence instead of manual Gary/operator confirmation.
- Which backend pytest failures are known environment blockers rather than product regressions.
- The operational interpretation of "world-class" as cognition/feedback/graph maturity rather than feature sprawl.

This should become structured operational memory.

## Fragmentation

- Wiki and app repo both describe production readiness, but the app repo has later phase evidence.
- Canonical architecture indexes exist but lag rapidly after each phase.
- Evidence is organized by phase, not by system capability.
- Rollback artifacts are stored as local paths in evidence docs, not indexed globally.
- PR bodies contain governance memory that is not always mirrored into docs.
- Some audit facts are machine-readable JSON; others are prose.

## Staleness Risks

- Wiki pages can describe staged/Supabase/retrieval states that predate Fortress-Prime canonicalization.
- App repo docs can mention old checker gaps even after production alignment.
- Worktree state and branch ancestry are not represented in the wiki as a live map.
- Evidence directories are excellent for auditability but weak for discoverability.

## What A New Engineer Would Miss

- Fortress-Prime, not `fortress-legal-app`, is the effective production implementation.
- Current work is stacked through phase branches and PRs, not simply on the protected base.
- The legal system is safe for controlled internal review only.
- The 232 unresolved source issues are intentionally visible and excluded.
- The checker is an operational gate, not just a smoke test.
- Most recent capabilities are in app repo operational docs rather than wiki context packs.

## What A New AI Session Would Miss

- Which worktree is clean and current.
- Which local dirty files are unrelated.
- Which evidence paths should be treated as authoritative.
- Which PR branch is the real base for the next phase.
- Which outputs may contain confidential legal text and must not be printed.
- Which repeated failures are known environment blockers.

## Required Evolution

- Add a canonical "current operational state" page that is updated by each phase.
- Build an operational memory index keyed by system, phase, branch, PR, evidence path, and verifier result.
- Convert recurring governance labels into machine-readable policy manifests.
- Convert rollback artifact references into an indexed rollback catalog.
- Create an AI session start packet generated from repo/wiki state rather than chat memory.
- Create knowledge graph nodes for systems, workflows, evidence, risks, blockers, and authority boundaries.
