## 3. regenerate_pdf_demos.py

**Location:** `backend/scripts/regenerate_pdf_demos.py`

**Invocation:**
```bash
.uv-venv/bin/python3 backend/scripts/regenerate_pdf_demos.py
# or as a module:
.uv-venv/bin/python3 -m backend.scripts.regenerate_pdf_demos
```

**What it does:**
1. Fetches real owner mailing addresses from Streamline's `GetOwnerInfo` endpoint
   (no DB writes — read-only API call)
2. Loads `Property` records from the local database (read-only)
3. Constructs in-memory `StatementResult` objects with synthetic balance/charge data
4. Calls `_build_pdf_bytes()` (the pure, sync rendering function) directly — no
   `OwnerBalancePeriod` or `OwnerPayoutAccount` rows are created in the DB
5. Writes each PDF to `crog_output/`
6. Writes a `.pdf.txt` companion file with verbatim extracted text
7. Self-verifies by checking all expected strings are present in the output

**Bug fixed:** `fetch_owner_info()` was checking `data.get("owner_id")` but
Streamline's `GetOwnerInfo` response uses key `"id"`. Fixed in
`backend/integrations/streamline_vrs.py`.

**Renderer refactor:** `statement_pdf.py` was refactored to extract `_build_pdf_bytes()`
— a pure synchronous function that takes all rendering data as explicit keyword
arguments with no database access. `render_owner_statement_pdf(db, period_id)` is now a
thin async wrapper that fetches from DB and calls `_build_pdf_bytes()`. All 33 Phase E
and Phase E.5 PDF tests pass unchanged after the refactor.
