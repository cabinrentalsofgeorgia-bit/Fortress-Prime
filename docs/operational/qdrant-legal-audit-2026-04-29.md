## §3 Throughput micro-benchmark — `legal-embed` via gateway

### §3.1 Method

- **Sample:** 100 documents pulled via Qdrant `points/scroll` from `legal_caselaw` (text field `text_chunk`).
- **Doc size:** min 604 chars, median 8,277 chars, **mean 5,929 chars**, max 9,744 chars. Caselaw has the largest chunks of any populated legal collection — this is therefore a **conservative (slow-end) baseline**.
- **Path:** `python3 → ssh admin@192.168.0.100 → curl http://localhost:8002/embeddings → LiteLLM gateway → spark-3:8102 NIM`. One request at a time, no concurrency, no batching, no keep-alive (each call spawns a new ssh + new curl).
- **Caller contract honored:** `model=legal-embed`, `input_type=passage`, `encoding_format=float`.

### §3.2 Results (99 of 100 succeeded)

```
n_attempted        : 100
n_ok               : 99
n_errors           : 1   (harness shell-quoting bug — see §3.4)

wall_clock_s       : 49.45    (includes ssh+curl spin-up per call)
sum_curl_s         : 17.21    (server-side time, summed)

server-side latency (curl --time-total, what the gateway actually saw):
  min   :  29 ms
  p50   : 203 ms
  mean  : 174 ms
  p95   : 307 ms
  p99   : 331 ms
  max   : 400 ms

end-to-end latency (Python perf_counter wrapping ssh+curl):
  p50   : 526 ms
  p95   : 628 ms
  p99   : 663 ms

throughput:
  wall-clock      : 2.00 docs/sec  (ssh-overhead dominated — NOT representative)
  server-side     : 5.75 docs/sec  (sequential, no batching)
```

### §3.3 Extrapolated reindex time per collection (server-side sequential, no concurrency, no batching — pessimistic floor)

| Collection | Points | Sequential wall-time @ 5.75 docs/s | Notes |
|---|---:|---:|---|
| `legal_caselaw`                  | 2,711   | **~ 7.9 min** | done in a coffee break |
| `legal_privileged_communications`| 241,167 | **~ 11.7 h**  | overnight job |
| `legal_ediscovery`               | 738,918 | **~ 35.7 h** | 1.5 days sequential — multi-shift, **must parallelize** |
| `legal_hive_mind_memory`         | 4       | <1 s         | trivial |
| `legal_library`                  | 3       | <1 s         | trivial |
| **Total (sequential floor)**     | 982,803 | **~ 47.5 h** | |

**Important caveats — these numbers are a floor, not a forecast:**

1. **Batching:** the OpenAI embeddings API accepts a list of inputs in one call. NIM almost certainly supports this; even modest batch=8 would 3–6× throughput depending on GPU saturation. Untested in this brief (out of scope per "100-doc benchmark is the only traffic" constraint).
2. **Concurrency:** sequential client. With even 4–8 concurrent workers feeding the gateway, throughput should rise sub-linearly until GPU bound. Untested.
3. **Doc size skew:** the 5.9k-char caselaw mean is the *largest* chunk size in the populated set. `legal_ediscovery` (~800 chars) and `legal_privileged_communications` (~800 chars) chunks are **~7× shorter**, which should embed faster per call (less prompt-token compute). The sequential extrapolation overestimates wall-time for those two collections by some unknown but likely meaningful factor.
4. **Network locality:** the bench drove the gateway from a Python runner *outside* spark-2, going through ssh. Real reindex code will run on spark-2 (or wherever the indexer lives) and will lose the ssh tax — but the server-side curl number already excludes most of that.

A realistic reindex job using batching + 4–8 workers should land **somewhere in the 2–6× faster range** than the sequential floor. Operator should treat the table above as "definitely feasible by this much time" — actual will be better.

### §3.4 The one error

```
i=52: bash: -c: line 1: unexpected EOF while looking for matching ``'
```

This is a **harness-side shell-quoting bug** — the doc payload contained backticks or another character that broke the inline JSON shell-escape in the `ssh ... 'curl ... -d "{...}"'` chain. It is **not** a gateway / NIM / model error; the next 47 calls after this one succeeded normally. Real reindex code that uses an HTTP client library (not shell-piped curl) will not hit this.

### §3.5 Quality / functional sanity reused from PR #300

PR #300 §9.7 cosine sanity already established that `legal-embed` distinguishes legal paraphrase from off-domain text (margin 0.754 vs nomic 0.547 on the same triplet, with `input_type=query`). No additional traffic generated here.

## §4 Recommendation for Phase A reindex PR scoping (operator decides A vs B)

Based on the throughput data:

### Option A — all populated legal collections in one PR

- Cost: ~47.5 h sequential floor → realistically **8–24 h** with batching + 4–8 workers, ran as a single long-running job.
- Pros: one cutover window; everything moves together.
- Cons: large job, large blast radius, harder to abort cleanly, blocks Phase B for the duration of the `legal_ediscovery` reindex (since that collection is the long-pole).

### Option B — caselaw + privileged in this PR; ediscovery in a follow-up PR

- This-PR cost: `legal_caselaw` + `legal_privileged_communications` = ~243k points → **~12 h sequential floor → ~2–4 h realistic** with batching + concurrency. Manageable overnight job.
- Follow-up PR: `legal_ediscovery` 738k points alone → ~36 h sequential floor → ~6–12 h realistic. Standalone job, can be scheduled when Phase B is paused or running off a frozen snapshot.
- Pros: smaller, faster-feedback first PR; isolates the Phase-B-blocking collection so it gets dedicated attention; matches the "DO NOT touch legal_ediscovery during Phase B" hard constraint without ambiguity.
- Cons: two cutovers; intermediate state where caselaw + privileged have moved to 2048-dim while ediscovery is still on 768-dim (callers / `legal_council.py` need to handle dim mismatch per collection during the gap, or the gap window must be short).
- The trivial collections (`legal_library`, `legal_hive_mind_memory`, `legal_headhunter_memory`) take seconds — fold them into whichever PR is convenient. `legal_hive_mind_memory` may need a separate model decision (its content is outcome labels, not document chunks — see §2).

**This audit recommends Option B** unless the operator has a stronger reason to bundle. The legal_ediscovery reindex is the clear long-pole and is the one Phase B is sensitive to; isolating it makes both halves cleaner.

## §6 Follow-up items / notes for reindex PR

- **Field-name branching:** reindex tooling must read `text_chunk` for `legal_caselaw` and `legal_library`, `text` for `legal_ediscovery` and `legal_privileged_communications`. `legal_hive_mind_memory` is a separate question (outcome labels, not chunks).
- **`encoding_format=float`** must be set on every embedding call (per PR #300 §9.5). LiteLLM/OpenAI default `None` is rejected by the NIM.
- **`input_type=passage`** is correct for indexing (asymmetric encoder). Retrieval calls in `legal_council.py` will need `input_type=query` — that's a separate, retrieval-side change.
- **Batch + concurrency:** the reindex PR should validate batched input (list-of-strings) is accepted by `legal-embed` via the gateway, before sizing concurrency. Even batch=8 with 4 workers should comfortably outpace the sequential floor used here.
- **Empty/sparse collections:** `legal_headhunter_memory` (0), `legal_library` (3), `legal_hive_mind_memory` (4) are candidates for `delete_collection` + recreate at the new dim rather than scroll-and-reembed — operator's call.
